AWS のニュースブログで、Aurora PostgreSQL の serverless を数秒で作れる `express configuration` が GA したという記事を見かけました。

https://aws.amazon.com/jp/blogs/aws/announcing-amazon-aurora-postgresql-serverless-database-creation-in-seconds/

特に良いなと思ったのが、VPC を自前で用意しなくても、そのまま立ち上げて試せる点です。

とりあえずやってみましたが、起動においてはべらぼうに早いですね。圧巻です。
体感 10 秒くらいで PostgreSQL が使える状態まで到達しました。

今回は、実際に触ってみた画面を見ながら、AWS の発表内容の補足も含めて軽く整理します。

## これまでの RDS 作成と何が違うのか

RDSは検証用途だとVPCとか諸々の初期設定が重いです。

今回の `express configuration` は、そのあたりをかなり割り切ってくれます。

AWS ブログの内容を噛み砕くと、ざっくり以下の思想です。

- まずはすぐ作って触れることを優先する
- 細かい設定は後から必要に応じて変えればよい
- 最初からインターネット経由で接続しやすくしておく
- 認証は IAM ベースをデフォルトにして、パスワード管理を減らす

この思想がかなり良くて、開発や検証の最初の一歩がかなり軽くなっています。
特に VPC を先に考えなくて良いのは、個人的にはかなり大きいです。

## 作成画面はかなりシンプル

まず、コンソールから作成する時点で項目がかなり絞られています。

![](https://storage.googleapis.com/zenn-user-upload/6bda25d97147-20260326.png)

各項目の鉛筆マークが記載されている物は、変更可能でした。例えば最大キャパシティなどです。

- DB エンジンは Aurora PostgreSQL 17
- DB インスタンスタイプは Serverless
- 最小キャパシティは `0 ACU`
- 最大キャパシティは `16 ACU`
- 非アクティブ後の一時停止は `300 秒`
- インターネットアクセスゲートウェイは有効
- 認証は IAM のみ
- VPC は無く、プライベートアクセス前提ではない

AWS ブログでも触れられていましたが、この構成では新しいインターネットアクセス用のルーティングレイヤーが使われていて、普段使っている開発ツールから PostgreSQL の wire protocol でそのまま接続しやすくなっています。

RDS を試したいだけなのに VPC やサブネット、ルート周りから考え始めるのは、辛いですし....

今回はその段差がかなり低くなっていて、とりあえず作って SQL を打つまでがとても短いです。

## 接続してみる

接続は IAM 認証トークンを使う形でした。

コンソールのコードスニペットをベースに、そのまま `psql` で接続しています。

![](https://storage.googleapis.com/zenn-user-upload/a7bcaafed958-20260326.png)

```bash
export RDSHOST="<your-hostname>"
psql "host=$RDSHOST port=5432 dbname=postgres user=postgres sslmode=require password=$(aws rds generate-db-auth-token --hostname $RDSHOST --port 5432 --region ap-northeast-1 --username postgres)"
```

接続後に試しに DB を 1 つ作ってみました。

```sql
CREATE DATABASE test_database;
```

ちゃんとそのまま作れています。

この流れがかなり良くて、作成、接続、SQL 実行までが 1 本の導線でつながっているので、検証環境としてかなり扱いやすいです。

## AWS ブログの補足

発表記事を読んだだけだと少し分かりづらい部分もあるので、実際に触った感覚で補足します。

### 1. これは本番向けの万能設定というより、始めるまでを速くする仕組み

最初の検証や PoC には向いているのかなと思います。

一方で、ネットワーク分離や細かい DB 設定を厳密に詰めたい本番環境では、そのまま何も考えず採用するものではないと思いました。

この「VPC を明示的に作らずに始められる」体験はかなり新しいものです。
2026年3月25日の AWS News Blog では、Aurora PostgreSQL の express configuration が GA したタイミングで、Aurora クラスターを VPC なしで作成できることと、新しい internet access gateway routing layer が導入されたことが案内されています。

一方で、Aurora の通常作成に関するドキュメントでは、今でも Aurora DB クラスターは VPC 内に作成する前提で説明されています。
なので理解としては、これまでの標準的な Aurora 作成は VPC 前提のままで、今回の express configuration から「まずは VPC を意識せずにすぐ立てる」導線が追加された、という認識でよさそうです。

### 2. IAM 認証がデフォルト

パスワードをSecretManagerなどで管理する運用を減らせるので、個人的にはこの方向はかなり好きです。

ただし、IAM 認証トークンは有効期限が短いので、接続ツールによっては都度取り直す必要があります。

ブログにもある通り、pgAdmin のようなツールを使う場合はトークンをパスワード欄に入れる形になります。

Kubernetesを用いている場合はIRSAなどを用いてIAM Policyをアタッチすれば簡単にRDS利用が可能になる点はやはり体験が良いですね。

AWSとしてはセキュリティの向上としてVPC内部に入れることを推奨はしているみたいなので、本番運用向けではないというのが正しい理解なのかなと感じました。

## Vercel 統合もかなり魅力的

2025年12月17日には、Vercel Marketplace から AWS Databases を直接作成できる統合も発表されています。

https://aws.amazon.com/jp/about-aws/whats-new/2025/12/aws-databases-are-available-on-the-vercel/

この統合では Aurora PostgreSQL、Aurora DSQL、DynamoDB を Vercel 側から選べます。

この体験はかなり魅力的で、DynamoDB のように気軽に作れる選択肢と同じ並びで PostgreSQL を選べるのは良いです。
Aurora PostgreSQL を使いたいけれど、最初のセットアップはなるべく軽くしたい、という用途にはかなり相性が良さそうです。

一方で、コスト感は DynamoDB ほど軽くはありません。

今回の作成画面では 1 ACU 時間あたり 0.15 USD と表示されていて、Serverless v2 の最小単位が 0.5 ACU ベースになる構成では、止まっていない時間の最低コスト感はそれなりにあります。
使ってないタイミングでは0までスケールインしてくれるとはいえ、動いている間のコストはそこそこ高めの印象です。

もちろん Aurora 側は PostgreSQL としての互換性や運用のしやすさがあるので単純比較はできませんが、Vercel と組み合わせて検証用にさっと使いたいだけなら、Supabase のような選択肢の方がコスパは良いケースもありそうです。

## どんな時に使いやすそうか

触ってみた感想としては、以下のようなケースにかなり向いていそうでした。

- 新しいアプリの検証用 DB をすぐ欲しい時
- AI コーディングツールやローカル実装からすぐ接続したい時
- Aurora をまず触ってみたい時
- ハンズオンやデモ用に短時間だけ DB を立てたい時

逆に、最初から厳密なネットワーク設計やセキュリティ境界を詰める必要がある案件では、いつもの構成で丁寧に作った方が安心だと思います。

## 削除はそれなりに時間がかかる

作成はかなり速いですが、削除は合計で 20 分程度かかりました。

すぐ作れるので気軽に触れる一方で、作ったあとにすぐ消えるわけではないので、この点は少し注意が必要です。

もっとも、RDS や Aurora を作って数分で消すこと自体がそこまで多くはないと思うので、実運用では大きな問題にはなりにくそうです。

## まとめ

今回は Aurora PostgreSQL の `express configuration` を試してみました。

結果としては、10 秒くらいで使える PostgreSQL が立つというのはかなり事実に近いです。

もともと、RDS は便利ですが、最初の一歩が少し重い印象もありました。
個人開発をするときにわざわざVPCやらサブネットやら、設定しないですし。それならDynamoで頑張るか。とかSupabaseでやるかという感じでした。

今回の機能はその重さをかなり削っていて、特に開発初期や検証用途ではかなり良さそうです。

しばらく触ってみて、料金感や再開時の待ち時間、普段の開発フローにどこまで組み込めそうかも見ていきたいと思います。
